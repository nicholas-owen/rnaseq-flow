Checking it — bugs found
Your pipeline was well-structured but had several bugs that would have broken real runs:
Show-stoppers

Nothing was ever published to results/ — no module had a publishDir directive and there was no modules.config, so every output stayed buried in work/.
rnaseq.nf referenced ch_multiqc_files ~120 lines before it was declared (crashes when STAR-Fusion runs); MultiQC was called without .collect() so it ran once per file instead of once over all files.
STAR-Fusion declared output filenames that didn't match what its script produced → process would fail.
STAR used --chimOutType WithinBAM, which never produces the Chimeric.out.junction file STAR-Fusion needs → fusions silently never ran.
HISAT2 piped into samtools that wasn't in its container, had no single-end support, and produced an unsorted BAM that samtools index then rejected.
rMATS input channel used .collect() which flattened BAM/BAI pairs into garbage; used a non-existent params.single_end.
IsoformSwitchAnalyzeR received Salmon results as [meta, dir] tuples instead of directories, and ran an R script via ${projectDir} that isn't visible inside the container.

Other fixes — gzipped reference handling (STAR/HISAT2/featureCounts/Salmon decoys/gtf2bed couldn't read .gz); R scripts loading packages absent from their containers (pheatmap, ggplot2, dplyr, rowVars); deprecated docker.userEmulation; Quarto's plotly dependency.
Ensuring it works — what was added

conf/base.config — real CPU/memory/time per process (every task was silently running on 1 core), capped by --max_*.
conf/modules.config — publishDir rules so results land in results/.
stub blocks on all 26 modules so the whole pipeline can be dry-run with -stub-run.
